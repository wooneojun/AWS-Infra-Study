## 네트워크 아키텍처 설계 시 가장 고려해야 할 VPC 연결 전략

현업에서 50개의 VPC 를 연결해야 하는 상황

1. 관리의 복잡성
2. 비용
3. 성능

3가지의 트레이드오프를 계산해야 한다.

## 1. VPC Peering (일대일 연결)

- 두 VPC 간의 직접적인 네트워크 연결
    
    ### 아키텍처 특성
    
    - 비전이성 라우팅이기 때문에 A-B, B-C 라고 해서 A-C가 연결되지 않는다
    - 모든 VPC 가 서로 통신하기위한 Full-Mesh 구조를 만드려면 N(N-1)/2 공식에따라 1225개의 피어링 연결과 라우팅 테이블 업데이트가 필요
        - 관리 오버헤드 폭증
    
    ### 장점
    
    - 대역폭 제한이 없고
    - Transit Gateway와 같은 추가적인 데이터 처리 비용이 발생하지 않음
    - 지연 시간 가장 낮음

## 2. AWS Transit Gateway (허브 앤 스포크)

- TGW 는 중앙 집중식 라우터 역할을 함
    
    ### 아키텍처 특성
    
    - 모든 VPC는 중앙 허브인 TGW에 연결되기만하면 서로 통신할 수 잇음
    - 50개의 연결만 생성하면 모든 VPC 가 서로 통신할 수 있음. 라우팅 관리가 획기적으로 단순히짐
    
    ### 장점
    
    - 확장과 관리의 유리함
    - 수백, 수천 개의 VPC 와 온프레미스 환경을 쉽게 통합할 수 있음
    
    ### 단점
    
    - TGW Attachment 시간당 유지비용
        - 서울리전 기준 연결당 $0.05/시간 (20시간당 1440원)
    - TGW 통과 트래픽에 대한 데이터 처리비용(GB당) 추가로 발생
        - 서울리전 기준 $0.02/GB
    - 각 VPC Attachment 의 최대 대역폭은 50Gbps 로 제한됨

## 실무 전략

- 전체 네트워크는 Transit Gateway 를 사용해 망을 구성하여 관리의 편의성을 챙기고
- 특정 VPC 간에 엄청난 양의 데이터 전송이 발생하거나 지연시간이 극도로 짧아야 하면 해당 두 VPC에만 예외적으로 VPC Peering 을 추가해 TGW 데이터 처리 비용을 아끼고 대역폭 제한 우회

**Q1. A, B, C 세 개의 VPC가 있습니다. A와 B는 VPC Peering으로 연결되어 있고, B와 C도 VPC Peering으로 연결되어 있습니다. A에 있는 EC2 인스턴스에서 C에 있는 RDS 인스턴스로 데이터를 전송하려고 하는데 통신이 되지 않습니다. 보안 그룹과 NACL은 모두 열려 있다고 가정할 때, 원인이 무엇이며 어떻게 해결해야 합니까?**

- VPC Peering 비전이성으로 A-C 가 직접 Peering 돼있지 않다면 통신할 수 없습니다. A-C VPC 간의 VPC Peering 을 연결하고 양쪽 VPC의 라우팅 테이블을 업데이트 하여 서로의 CIDR 블록으로 향하는 트래픽이 Peering 커넥션을 타도록 해야 합니다. A 에 있는 EC2 인스턴스와 C에 있는 RDS 간의 통신이 가능해질 것입니다.

**Q2. 회사의 클라우드 환경에 100개의 VPC가 Transit Gateway로 연결되어 운영 중입니다. 이번에 빅데이터 처리용 VPC(VPC-Data)와 대규모 머신러닝 학습용 VPC(VPC-ML) 두 개를 새롭게 구축했습니다. 이 두 VPC 간에는 하루에 수십 테라바이트(TB)의 데이터 이동이 예상됩니다. 기존처럼 Transit Gateway에 이 두 VPC를 연결하는 것만으로 충분할까요? 아키텍트 관점에서 비용과 성능을 고려한 최적의 라우팅 설계는 무엇입니까?**

- Transit Gateway는 많은 VPC 간 네트워크 연결을 허브엔스포크 방식으로 관리의 편의성을 제공한다는 장점이 있지만 대역폭의 제한과 전송당 비용이라는 단점도 포함하고 있습니다. 하루 수십 테라바이트의 데이터 전송은 TGW 의 50Gbps로 충분하지 않을 가능성이 있고 데이터 전송 비용을 고려해야 합니다. 관리의 편의성을 위 해 TGW 를 기본으로 쓰되 Heavy Traffic 구간에만 VPC Peering을 사용하는 하이브리드 라우팅 전략을 사용하겠습니다. 빅데이터 처리용 VPC 와 대규모 머신러닝 학습용 VPC 는 VPC Peering 연결을 하는 아키텍처를 사용하게 되면 각 연결방식의 장점을 적합하게 사용할 수 있을 것입니다.
- 라우팅 테이블에는 보통 0.0.0.0/0 → tgw-xxx로 잡혀있는데 VPC-Data 와 VPC-ML 간의 통신이 TGW가 아닌 Peering을 타게 하려면 라우팅 테이블에 상대방 VPC 의 특정 CIDR 대역폭을 지정해 Peering 커넥션으로 향하는 경로를 추가해야 합니다. 가장 긴 접두사 일치 원칙에 따라 0.0.0.0/0 보다 긴 접두사를 가진 IP 대역(172.16.0.0/16 -> pcx-12345678)이 우선순위가 가지게 됩니다.

**Q3. Transit Gateway를 사용하면 관리 비용이 줄어든다고 하지만, 인프라 사용 비용 측면에서는 VPC Peering과 다른 과금 체계가 적용됩니다. Transit Gateway 도입 시 고려해야 할 '추가적인 과금 요소' 두 가지는 정확히 무엇입니까?**

- TGW 통과 트래픽 당 데이터 처리비용과 TGW Attachment 시간당 유지비용이 있습니다.

## 주의할 점

- IP 대역이 겹치는 상황(Overlapping CIDRs) 은 클라우드 환경에서 M&A 로 인해 다른 회사의 인프라를 통합하거나 각 개발팀이 아무 생각 없이 10.0.0.0/16으로 VPC를 찍어냈을 때 발생
    
    ### VPC Peering의 경우
    
    - 생성 자체가 불가능
    - CIDR 대역에 단 1개의 IP 라도 겹치는 두 VPC 간에 Peering 생성을 막는다. 라우팅 테이블에서 패킷이 어디로 가야할지 알 수 없는 라우팅 모호성이 발생하기 때문
    
    ### TGW 의 경우
    
    - 연결은 되지만 겹치는 대역간의 통신은 꼬인다.
    - TGW 라우팅 테이블에 10.0.0.0/16의 목적지가 두 개가 되므로 패킷이 A 로 가야할지 B로 가야할 지 몰라 통신이 실패하거나 엉뚱한 곳으로 간다.

## 겹치는 대역 해결책

1. 베스트 케이스 (사전예방) : AWS IPAM (IP Address Manager)을 도입해 사내 모든 VPC CIDR 할당을 중앙 통제하여 애초에 안겹치게
2. Private NAT Gateway 활용 : TGW 라우팅 테이블과 Private NAT Gateway 를 활용해 출발지/ 목적지 IP 를 속이는 복잡한 구성을 짜야한다. 관리 포인트가 늘어나 매우 힘든 경우
3. AWS PrivateLink (VPC Endpoint Services) 사용 : 라우팅 테이블을 직접 건들지 않고 제공자의 VPC 에 서비스를 띄우고 소비자의 VPC 에는 해당 서비스와 1:1로 매핑되는 프라이빗 IP(ENI 가상 랜카드) 를 하나 꽂아둔다.
    1. IP 대역 전체를 연결하는 것이 아니라 특정 애플리케이션 포트 간의 단방향 통신만 허용하므로 CIDR 대역이 완전히 겹쳐도 문제없이 안전하게 통신할 수 있음
