## 부하 테스트 도구 비교

| 구분 | JMeter | k6 | nGrinder | Locust |
| --- | --- | --- | --- | --- |
| 개발 언어 | Java | Go | Java | Python |
| 사용 방법<br>(스크립트 작성 언어) | GUI, CLI<br>XML 작성, Java로 실행 | GUI, 주로 CLI<br>JavaScript 코딩 | GUI, CLI<br>Groovy, Jython 코딩 | CLI<br>Python 코딩<br><br>Web UI<br>모니터링 |
| 리소스 & 동시 접속자 수 | 스레드 기반 모델이라 스레드 할당에 상당한 리소스 필요 | 고루틴 기반<br>비동기 I/O로 리소스 소모가 매우 적음 | 스레드/프로세스 기반 | gevent 기반 비동기 이벤트 코루틴 사용<br>이벤트 및 비동기 접근 방식 기반 모델<br>JMeter보다는 가볍지만 대규모 테스트 시 리소스가 많이 필요 |
| 특징 | - 20년 이상의 **방대한 레퍼런스**<br>- **다양한 프로토콜** 지원<br>- 풍부한 사용자 인터페이스<br>- 다양한 테스트 요소와 보고서 결과 분석 기능 제공<br>- **실제와 비슷한 분산 부하 테스트** 가능<br>- 실시간 모니터링 및 스케일링은 CLI 모드와 **플러그인**으로 일부 확장 가능 | - **간단하고 직관적인 API**를 통해 스크립트 작성 가능<br>- 테스트 실행 중 실시간 모니터링 제공<br>- 분산 부하 테스트는 k6 Cloud 서비스를 통해 제공<br>- **CI/CD 친화적** | - 다양한 프로토콜 지원<br>- 분산 테스트 실행 가능<br>- 실시간 모니터링 제공<br>- 한국어 지원<br>- Naver에서 개발 | - 실시간 모니터링 및 대시보드를 통해 부하 테스트 결과와 성능 지표 확인 가능<br>- 실제와 비슷한 분산 부하 테스트 가능<br>- Python 코드로 유연한 시나리오 작성 가능 |
| 장점 | 1. **코딩 불필요**<br>2. **프로토콜 범용성**<br>3. **커뮤니티와 플러그인** | 1. 가장 **가볍고 빠름**<br>2. **개발자 친화적**<br>3. **CI/CD 통합** | 1. **국내 레퍼런스**가 많고 배우기 쉬움<br>2. **웹 기반 통합 플랫폼**<br>3. **Java 생태계 호환성** | 1. **Python 환경**<br>2. **Web UI 제공** |
| 단점 | 1. **무거운 리소스** 소모<br>2. **버전 관리의 어려움** | 1. GUI 부재<br>2. **프로토콜 제한** | 1. 인프라 구축 부담<br>   - 웹 서버와 부하 생성 서버를 각각 설치 및 세팅해야 함<br>2. 업데이트 주기가 김 | 1. Python GIL 제약<br>   - 대규모 부하 테스트를 위해 여러 워커 노드로 구성한 분산 환경이 필요 |
| 사용 사례 | - DB 쿼리 성능이나 TCP 통신처럼 웹 외의 다양한 프로토콜 테스트가 필요한 경우<br>- GUI 기반 도구가 필요한 경우 | - 최소한의 서버 장비로 극단적인 대규모 트래픽을 발생시켜야 하는 경우<br>- MSA 환경에서 CI/CD 배포 파이프라인에 성능 테스트를 자동화해 포함하고 싶은 경우 | - QA팀, 개발팀, 기획팀이 하나의 웹 화면을 보며 테스트 결과를 공유하고 소통해야 하는 국내 기업<br>- Java/Spring 기반 서비스를 운영 중이고 복잡한 사내 비즈니스 로직을 부하 테스트에 반영하고 싶을 때 | - 팀의 주력 언어가 Python이거나 AI/데이터 분석 서버의 성능을 테스트할 때<br>- 로그인 → 특정 데이터 추출 → 조건문 분기 → 동적 요청 같은 복잡한 유저 시나리오를 코드로 구현해야 할 때 |
