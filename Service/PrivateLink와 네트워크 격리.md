데이터가 인터넷 구간을 타지 않고 안전하게 통신하게 만드는 기술

내부 망(프라이빗 서브넷)에 있는 EC2 나 Lambda가 S3, DynamoDB 같은 AWS 퍼블릭 서비스나 서드파티 SaaS 와 통신하려면!

- NAT Gateway 를 타고 인터넷으로 나가는 것은 보안상 위험하고 NAT 데이터 처리 비용도 많이 든다

<aside>
💡

VPC Endpoint 가 필요

</aside>

1. Gateway Endpoint (무료, 라우팅 테이블 기반)
    - 지원 서비스 : S3 와 DynamoDB 두개
    - 동작 방식 : VPC 의 라우팅 테이블에 접두사 목록(Prefix List)를 추가해 트래픽을 AWS 내부 네트워크로 바로 라우팅함
2. Interface Endpoint (유료, ENI 기반 - AWS PrivateLink)
    - 지원 서비스 : SQS, SNS, Kinesis, KMS 및 수많은 서드파티 SaaS
    - 동작 방식 : Private Subnet 내부에 프라이빗 IP 주소를 가진 가상 랜카드(ENI) 를 꽃아두고 DNS 이름을 해석하면 프라이빗 IP 로 연결되어 AWS 내부망을 통해 안전하게 통신

**Q1. 당신은 금융권 프로젝트의 보안 아키텍트입니다. Private Subnet에 있는 대규모 로그 수집용 EC2 인스턴스들이 매일 엄청난 양의 로그를 S3 버킷으로 전송하고 있습니다. 현재는 NAT Gateway를 통해 S3와 통신 중인데, NAT Gateway 데이터 처리 비용이 너무 많이 나와서 이를 해결하라는 지시를 받았습니다. 비용을 '0원'으로 만들면서 보안은 강화할 수 있는 아키텍처 변경안을 제시해 주세요.**

- VPC 게이트웨이 엔드포인트를 사용해 VPC 라우팅 테이블에 접두사 목록을 추가하여 트래픽을 AWS 내부 네트워크로 바로 라우팅 합니다. 이 경우 내부 네트워크만 사용하므로 보안이 강화됨과 동시에 비용도 0원으로 사용 가능합니다.
- Gateway Endpoint 를 사용하면 기본적으로 VPC 내의 모든 리소스가 모든 S3버킷에 접근할 수 있게 됩니다. 이를 방지하기 위해 엔드포인트 정책을 작성하여 특정 S3버킷으로만 데이터를 보낼 수 있도록 최소 권한 원직을 적용하겠습니다.
    1. VPC 엔드포인트 정책 (나가는 문 단속)
        1. 주체 : VPC 엔드포인트
        2. 목적 : VPC 안에서 이 엔드포인트를 타고 밖으로 나갈 때 목적지가 어디인지 통제
        3. 작동 방식 : 기본값 FullAccess - 이를 제한해 이 엔트포인트를 통과하는 트래픽은 오직 my-company-log-bucket 라는 이름의 S3 버킷으로만 갈 수 있다고 명시
    2. S3 버킷 정책 (들어오는 문 단속)
        1. 주체 : S3 버킷 리소스
        2. 목적 : 내 버킷으로 들어오는 데이터가 어디서 출발 했는지 통제
        3. 작동 방식 : S3 버킷에 정책을 걸 때 Condition 절에 aws: sourceVpce 또는 aws:sourceVpc 조건을 명시 

**Q2. 회사의 핵심 서비스가 A 계정의 VPC에 있고, 보안 감사를 위한 모니터링 솔루션(서드파티)이 B 계정의 VPC에 있습니다. 두 VPC의 IP 대역(CIDR)이 겹쳐서(Overlapping) VPC Peering이나 Transit Gateway로는 연결할 수 없는 상황입니다. 트래픽을 인터넷으로 내보내지 않고 두 서비스 간에 단방향(A -> B) 통신을 안전하게 연결하려면 어떤 기술을 사용해야 하며, 그 이유는 무엇입니까?**

- ~~A의 VPC에 서비스를 띄우고 AWS PrivateLink를 활용해 B계정의 Private Subnet에 ENI 를 만들어 이 주소로 단방향 통신을 수립합니다. IP 대역이 곂치는 경우 VPC Peering 이나 TGW를 사용하기 어렵기 때문에 AWS PrivateLink를 사용하면 이런 상황을 회피할 수 있습니다.~~
- PrivateLink 는 소비자가 공급자를 호출하는 구조
    - 공급자 B 계정의 VPC 에 NLB 를 띄우고 이를 기반으로 VPC 엔드포인트 서비스를 생성해야 함
    - 소비자 A 계정의 VPC에 ENI 를 생성하여 B계정의 서비스와 연결
    - 트래픽 흐름
        - A 계정의 EC2 → A 계정의 ENI(프라이빗 IP) → AWS 내부망 → B계정의 NLB → B 계정의 모니터링 서버
        - A에서 B 는 접근 가능하지만 B 에서는 A의 네트워크로 침투할 수 없음
