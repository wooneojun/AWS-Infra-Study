- 클라이언트 to 벡인드 서비스를 모아주는 단일 진입점
- 고가용성 완전 관리형 서비스
- 크게 3종류 제공됨

# 1. API 게이트웨이의 주요 역할

- **라우팅** : /users 요청은 사용자 서비스로, /orders 요청은 주문 서비스로 교통정리 해줌
- **인증 및 인가(Auth)** : 유효한 사용자인지 API 를 호출할 권한이 있는지 검사
- **속도제한(쓰로틀링)** : 특정 사용자가 너무 많은 요청을 보내지 못하게 차단
- **프로토콜 변환** : 클라이언트 HTTP 요청을 백엔드가 이해할 수 있는 다른 방식(Lambda 호출, gRPC) 등으로 변환
- **부하 분산 및 캐싱** : 게이트웨이에서 응답을 캐싱함

# 2. AWS API Gateway의 3가지 종류

## 2.1 HTTP API

- 가장 최근에 나온 가성비 및 성능 중심
- **역할** : Lambda나 HTTP 백엔드로 요청을 단순히 전달하는데 최적화 돼있음
- **장점** : REST API 보다 최대 70% 저렴하고 지연 시간이 매우 낮음. OIDC, OAuth 2.0 인증을 기본 지원
- **용도** : 단순한 마이크로서비스 연결, 서버리스 백엔드 구축 등 성능과 비용 중요할 때

## 2.2 REST API

- 가장 오래된 정석적인 API게이트웨이
- **역할** : API 키 관리, 요청/응답 변환, 사용량 계획 등 복잡한 관리 기능이 필요할 때 사용
- **장점** : 캐싱 기능을 세밀하게 조절할 수 있고 AWS WAF 와 연동해 보안 강화, API를 외부에 상품화하여 팔 때 유용
- **용도** : 보안 요구사항이 까다롭거나 클라이언트마다 다른 호출 한도를 설정해야하는 복잡한 기업용 서비스

## 2.3 WebSocket API

- 연결 지향형 실시간 통신
- **역할** : 클라이언트와 서버가 언제든 데이터를 주고 받을 수 있도록 통로를 계속 열어둠
- **장점** : 서버가 클라이언트에게 먼저 데이터를 보낼 수 있음
- **용도** : 채팅 앱, 실시간 주식 차트, 멀티 플레이어 게임, 실시간 대시보드 

| **특성** | **HTTP API** | **REST API** | **WebSocket API** |
| --- | --- | --- | --- |
| **주요 목적** | 고성능, 저비용 | 풍부한 기능, 관리 | 실시간 양방향 통신 |
| **비용** | 매우 저렴 | 보통 | 연결 시간 및 메시지당 과금 |
| **지연 시간** | 매우 낮음 | 보통 | 실시간 |
| **인증 방식** | OIDC, OAuth 2.0 | API 키, IAM, Cognito | IAM, Lambda Authorizer |

# 3. 내부구조

## 3.1 인프라 레벨

- AZ 분산 : 내부적으로 AWS가 Multi AZ 에 걸처 리소스를 자동으로 분산 배치한다. 한 데이터 센터가 장애가 나도 다른 곳에서 즉시 처리
- 리전별 엔드포인트 : API 게이트웨이는 특정 리전 내에서 동작하며 해당 리전 전체가 마비되지 않는 한 서비스 유지됨

## 3.2 SPOF 방지 전략

### 3.2.1 멀티리전 배치

- 여러 리전에 걸쳐 API 게이트웨이를 배치한다.
- Route 53 연동 : Route53의 상태 확인 기능을 사용해 서울 리전이 응답하지 ㅇ낳으면 자동으로 도쿄 리전으로 트래픽을 돌림

### 3.2.2 쓰로틀링 및 할당량 설정

- 특정 클라이언트가 요청을 독점하면서 논리적 SPOF 가 발생할 수 있음
- Usage Plan을 통해 클라이언트 당 초당 호출 수를 제한하여 전체 시스템을 보호

### 3.2.3 서킷 브레이커 패턴 도입

- 백엔드서비스가 죽었는데 API 게이트웨이가 계속 요청을 보내면 대기열이 쌓여 API 게이트웨이에 영향을 줄 수 있음
- **서킷 브레이커 패턴** : 전기 회로 차단기에서 유래한 패턴으로 망가진 서비스에 계속 요청을 보내서 시스템 전체가 무너지는 것을 방지
    - API게이트웨이가 오류상태인 백엔드 에 요청을 보낼 때 응답이 오기까지 쓰레드를 점유하게 되고 이게 쌓이면 API 게이트웨이 전체가 마비된다
    - 이를 장애 전파(Cascading Failure) 라고 한다.
    - **3가지 상태**
        - **Closed(정상)** : 모든 트래픽을 통과시킴
        - **Open(차단)** : 에러 발생률이 설정한 임계치를 넘으면 회로를 끊음. 이후 요청은 백엔드에 묻지도 않고 즉시 에러를 반환
        - **Half-Open(반개방)** : 일정 시간이 지나면 일부 요청만 보내보고 성공하면 Closed, 실패하면 Open
- 백엔드 상태가 이상하면 API Gateway 단에서 즉시 에러를 반환하여 백엔드가 회복할 시간을 벌어줌
- AWS 서비스 메시 도구인 App Mesh를 사용하면 코드 수정 없이 서킷 브레이커를 설정할 수있음
    - 인프라 설정만으로 구현 가능하며 정교
- Lambda + Step Functions (서버리스) 로 장애시 흐름을 제어
    - 서버리스 환경에 최적화
- 라이브러리 사용
    - 코드 내부에서 Resilence4j나 Hystrix같은 라이브러리를 사용해 백엔드 서비스 코드가 스스로를 보호
    - 각 마이크로 서비스마다 코드를 추가해야 하므로 운영오버헤드 증가

### 3.2.4 WAF 연동

- 디도스 공격이나 악의적인 요청으로 게이트웨이가 마비되는 것을 막음
- L7레이어의 패턴과 행동을 본다
    - Rate-based Rules : ‘특정 IP 가 5분간 100번 넘게 요청하면 차단’같은 규칙 적용
    - IP Reputation: AWS 가 보유한 전세계 악성 IP 리스트기반 차단
    - Bot Control : 요청의 헤더나 행동 패턴을 분석해 봇처럼 보이면 차단
