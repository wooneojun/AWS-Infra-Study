## 무중단 배포

- 배포 중에도 트래픽은 항상 정상 상태의 인스턴스로만 전달디고 LB 와 헬스체크를 통해 배포 중 문제가 생기면 자동으로 우회하거나 롤백

### 1. 롤링 배포

- 기존 서버 하나를 새 버전으로 교체해보고 이상이 없으면 순차적으로 다른 서버도 바꾸는거
- 단위 : 서버 하나씩

| 장점 | 단점 |
| --- | --- |
| 무중단 배포 가능 | 배포 중 새버전과 구버전 혼재되어있음 |
| 리스크 분산 안정적 | 장애 발생 시 롤백에 시간과 절차 필요 |
| 별도 환경 필요 없어 리소스 효율적 |  |
- 사용 예시 : 일반적인 웹 서비스

1. 버전 혼재로 인한 호환성
    - API 하위호환 유지
        - 기존 기능을 지원하면서 새로운 기능을 추가
    - 기능 스위치 적용
        - 배포중 장애가 있을때 자동 롤백되도록 설정

### 2. 기능 스위치

- 새로운 기능을 코드에 미리 넣어두고 실제 실행 여부는 스위치에 따라 제어
- 기능을 단계적으로 사용자에게 제공하거나 문제 발생했을 때 즉시 비활성화 가능
- 새로운 기능을 미리 코드에 넣어두고 실행 여부는 스위치가 제어

```python
#해당유저한테 신기능을 켜줄지 말지를 판단
if is_feature_enabled("recommendation_v2", user_id): 
    return new_recommendation()
else:
    return old_recommendation()
```

| 장점 | 단점 |
| --- | --- |
| 배포 후에도 기능 제어 유연성 | 복잡한 로직 필요 |
| 실시간으로 기능 제어 가능해 문제 발생 시 빠르게 대응 가능 | 기능이 예상보다 빨리 활성화되거나 비활성 안되는 문제 |
- 사용 예시  : CI/CD 환경에서 점진적 기능 활성화할 때 사용
- 실험적인 기능 배포하면서 사용자 피드백 받는 서비스

### 3. 블루/그린 배포

- 현재 운영중인 환경과 새 버전환경을 따로 만들고 그린이 준비되면 로드밸런서를 전환해 전체 트래픽을 새 버전으로 넘김

| 장점 | 단점 |
| --- | --- |
| 배포 실패 시 빠르게 롤백 가능 | 리소스 소비가 큼 |
| 새 환경에서 사전 테스트 가능 | DB 마이그레이션 시 동기화 문제 발생 가능 |
- 사용 예시 : 중대한 기능 변경이 포함된 서비스

<aside>
💡

발생할 수 있는 문제와 해결법

</aside>

- DB 구조 차이 → 점진적 마이그래이션, 이전 구조와의 동시 지원
- 리소스 비용 증가 → 배포 후 비활성 환경 자동 제거 스크립트 적용

### 4. 카나리 배포

- 새 버전을 먼저 일부 사용자나 서버에만 배포하고 이상이 없으면 점차 전체로 확대하는 방식
- 초기 문제를 빠르게 확인하고 전체 사용자에게 영향을 주기 전에 수정할 수 있음
- 롤링 배포는 모든 사용자가 신버전을 볼 수 있지만 카나리 배포는 특정 사용자 그룹을 나눠 먼저 새 버전을 사용하게 하는 사용자 단위의 배포 방식

| 장점 | 단점 |
| --- | --- |
| 장애 발생 시 피해 범위 최소화 | 초기  대상 사용자 선정 및 트래픽 분배 전략 필요 |
| 사용자 반응 기반 점진적 확대 가능 | 모니터링과 자동화 없으면 의미 없음 |
- 사용 예시 : 실시간 위험 감지 및 피드백이 중요한 서비스 (예: 추천 알고리즘 변경, 결제 로직 변경, 대규모 게임 서버, 금융거래 시스템 등등)

<aside>
💡

발생할 수 있는 문제와 해결법

</aside>

- 문제 발생시 대응 지연 → 실시간 모니터링 및 알림 시스템 필수
- 사용자 간 버전 불일치 혼란 → 내부 사용자 대상 선적용 후 점진 확대

### 5. 섀도우 배포

- 새 버전의 애플리케이션 배포 후 실시간으로 트래픽을 기존 시스템과 새 시스템에 동시 전달하는 방식
- 새 시스템에서의 트래픽은 실제 처리되지 않고 로그를 기록하거나 대시보드로만 모니터링 하여 실제 사용자에게 영향을 미치지 않으며 새 시스템이 잘 작동하는지 검증

| 장점 | 단점 |
| --- | --- |
| 사용자에게 영향 주지 않으면서 새 버전이 실제 트래픽 처리가능한지 확인 가능 | 추가적인 리소스 필요, 시스템 부하 더 커질 수 있음 |
| 고객 불만 없이 시스템 성능 점검 가능 | 모니터링과 로그 분석이 중요 |
- 사용 예시 : 실시간 트래픽 검증이 필요한 서비스 (예: 대규모 스트리밍 서비스, 금융 서비스 등)

## 비무중단 배포 방식

### 1. 재시작(Recreate)

- 다 중단하고 새로운 버전으로 한번에 교체
- 배포중 서비스 중단 있지만 배포 완료되면 서비스 새 버전으로 시작

| 장점 | 단점 |
| --- | --- |
| 버전 혼합 문제 없음 | 장애 발생 시 사용자 불편 |
| 배포 과정 간단하고 구현 난이도 낮음 | 롤백 불가능 |
- 사용 예시 : 소규모 시스템이나 서비스 중단 시간이 중요한 문제가 아닌경우

### 2. A/B 배포

- 기존 버전과 새 버전을 동시 배포하고 사용자 트래픽을 두 가지 버전으로 나누어 일부 사용자는 기존 버전을 나머지는 새 버전을 사용하게하는 방식
- 두 버전의 성능을 비교하고 최적의 버전을 선택할 수 있게 된다

| 장점 | 단점 |
| --- | --- |
| 사용자 반응 기반 새 버전 평가 가능 | 트래픽 나누는 전략 분석 복잡, A/B 테스트 결과 정확해야 함 |
| 성능차이를 비교해 최적화된 배포 전략 찾기 가능 | 사용자 혼란 초래 가능 |
- 사용 예시 : 대규모 플랫폼에서 사용자 경험이나 UI/UX 테스트 시 사용
